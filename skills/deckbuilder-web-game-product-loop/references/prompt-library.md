# Prompt Library

These prompts are reusable scaffolds. They work because they request evidence, decisions, files, and verification—not because of magic wording. Replace bracketed fields and attach the actual repository, screenshots, or video.

## P01 — Original vertical slice from a genre reference

```text
我想做一个受[参考游戏]核心循环启发、但名称、美术、文本、数值和代码全部原创的网页游戏。

主题：[原创主题]
平台：电脑浏览器 + 手机横屏
交付：无需安装，打开即可玩；最终需要公开链接

先不要追求完整商业体量。请冻结一个能打通的垂直切片：地图 → 战斗 → 奖励 → 下一站 → 首领 → 结算。
先给出本轮明确范围、不会做的内容和可验证的完成标准，再实现。主操作必须在一个屏幕内可见。
```

## P02 — Reference video as interaction evidence

```text
请观察我提供的完整玩法视频。不要只总结画风，要按时间证据拆解：
1. 玩家每一步输入是什么；
2. 游戏如何显示可选目标、敌人意图和结算；
3. 战斗、地图、奖励如何切换；
4. 哪些是可以借鉴的交互结构，哪些内容必须原创；
5. 当前项目最应该先改的三项。

分析后先复述你理解的操作序列，再改代码，避免把点击误做成拖拽。
```

## P03 — Interaction contract correction

```text
当前实现错在：[实际错误]。
正确操作应该是：[逐步输入序列]。

请把它写成明确状态机，包括选中、目标高亮、确认、无效目标、取消和多目标情况；然后实现并补回归检查。不要只改动画表面。
```

## P04 — Screenshot diagnosis

```text
这是玩家截图。请先只根据可见证据判断：
- 哪一层仍正常；
- 哪一层缺失/遮挡/裁切；
- 更可能是布局、资源、浏览器兼容、缓存还是玩家操作；
- 如何用最小实验区分这些原因。

修复后请用[目标设备/视口]和相邻断点复测，不要只测试截图的一个宽度。
```

## P05 — One-screen responsive requirement

```text
游戏必须在一个横屏视口内显示主要战斗信息和主操作。请按可用高度而不只是宽度设计。

至少检查：桌面、1366×768、844×390、780×430、568×320，以及每个媒体查询边界的前后1px。主操作不能被手牌、浏览器栏或安全区域挡住；触控目标至少约44×44；不得出现遮挡卡牌的纵向滚动条。
```

## P06 — Product differentiator

```text
现在它还是“[参考游戏]换皮”。请提出一个与[主题]天然相关、会改变玩家决策的第二构筑轴。

不要只给数值加成。它必须包含：可见载体、获得方式、升级/组合、定向构筑、信息说明、战斗反馈和至少一个与卡牌互动的规则。先说明玩家每回合会因此多考虑什么，再实现最小可玩版本。
```

## P07 — Story with downstream consequences

```text
加入故事系统，但不要把选择做成立即领取奖励。

每个主要选择至少要有：
1. 当下结果；
2. 后续章节回调；
3. 一个可观察的玩法影响；
4. 对首领/结局/档案中的至少一项影响。

区分单局旗标和跨局故事档案，并提供旧存档迁移测试。
```

## P08 — Licensed scene audio shortlist

```text
先定义需要循环配乐的场景和只需要短音效的动作，再从多个合法平台筛候选。

每个场景给5个候选，列出曲名、试听链接、许可证、是否允许商用/修改、是否要求署名、循环适配、移动端文件体积风险和你的判断。优先CC0/CC BY；不要把“免费下载”等同于可商用。由我选择后再下载、压缩、接入和写鸣谢。
```

## P09 — Adversarial review

```text
不要再证明它能打开。请把自己当成想让这次发布失败的测试员，从当前代码和实际游戏中找出10个不同的问题。

每个问题必须包含：复现场景、玩家影响、代码/状态/截图证据、根因、最小修复、通过标准。优先查状态迁移、重复动作、旧存档、断点边界、移动端浏览器栏、奖励构筑、路线真假、内容重复和公开缓存。
```

## P10 — Sequential issue closure

```text
把这10个问题逐项解决，不要一次性做成一个无法定位回归的大改。

对每一项：先写失败条件 → 实现最小修复 → 跑该项测试 → 跑兼容性测试 → 记录结果。全部完成后再做公共链接验收。只在10项都有独立证据、旧存档可用、公开链接指向新版本后停止。
```

## P11 — Public release without breaking active users

```text
发布到现有公开链接，不能让正在玩的用户突然失效。

请保留旧入口或做兼容跳转，升级存档结构时迁移旧数据；给CSS/JS加版本标识；推送后确认部署对应目标commit；从公网抓取HTML、JS、CSS和代表资源，验证版本与关键功能签名。最终只给“直接游玩链接”，另附仓库链接但不要混淆两者。
```

## P12 — Package the conversation into a reproducible skill

```text
不要只总结流程。请从聊天记录、仓库、提交和测试中提取：
- 可观察的长期偏好和决策；
- 真实失败案例及其根因；
- 可复制提示词；
- 架构模式和验收门槛；
- 不能声称知道的隐藏推理边界。

封装成SKILL.md + references + scripts + evals。然后在无本对话历史的新会话里做有Skill/无Skill对照、同提示词重复采样和异主题类比，报告均值、方差、失败项和是否达到可复用水平。
```

## Completion report template

```text
结果：[玩家现在能做什么]
入口：[直接链接/文件]
版本：[版本/commit]

本次改变：
- [行为1]
- [行为2]

验证：
- [逻辑测试]
- [响应式/视觉检查]
- [旧存档/兼容]
- [线上资源和部署]

仍未验证：[没有就写无；不要用静态检查冒充真实浏览器检查]
```
