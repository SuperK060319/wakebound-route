## 一、产品定义

《沙海机城》是一款原创横屏浏览器牌组构筑 Roguelike。玩家驾驶一座依靠巨型履带迁徙的机械城，穿越不断重组的沙漠遗迹，在能源耗尽前抵达“天穹钻机”。

核心体验：

> 观察敌方行动 → 编排卡牌顺序 → 驱动城环模块 → 承受代价或制造联动 → 选择路线和奖励 → 调整牌组与机城结构 → 击败首领。

平台与交付：

- 电脑浏览器、手机横屏。
- 点击操作，不依赖拖拽、右键或悬停。
- 无需安装，公开 URL 打开即玩。
- 单局目标时长：首版 15～25 分钟。
- 名称、世界观、角色、卡牌、图形、文本、音效和数值全部原创。
- 不采用题目禁止的主题词汇、意象和材质语言，也不复刻任何现有爬塔游戏的界面构图、角色轮廓、卡面或文案。

视觉方向：高温沙漠、象牙白陶瓷装甲、黑曜石机械、琥珀色能源、几何遮阳结构和巨型履带城市。避免通用废土拼贴感。

```text
标题页
  → 新游戏/继续游戏
  → 生成带种子的分支路线
  → 普通战斗
  → 卡牌或城环模块奖励

  → 工坊/遗迹事件
```

### 首版内容冻结

- 1 座可操控机城：“逐日城”。
- 1 个初始牌组：8 张。
- 卡牌池共 12 张：攻击5、防护4、调度3。
- 3 种普通敌人，组成2套双目标遭遇。
- 1 个两阶段首领：“天穹钻机”。
- 6 个城环模块，单局最多安装3个。
- 1 个工坊节点、2个简短遗迹事件。
- 4层真实分支地图；每局经过2场普通战斗和1场首领战。
- 生命、护甲、行动点、抽牌堆、弃牌堆、敌方意图、异常状态。
- 战斗后奖励、删牌或改造、胜负结算、保存和继续游戏。
- 固定种子模式，便于复现测试。

首版明确不做：多角色、完整章节剧情、在线账户、排行榜、多人模式、大规模跨局天赋树、复杂动画皮肤和付费系统。

## 三、原创双构筑系统：牌组 × 城环编排

### 1. 第一构筑轴：牌组

每张卡除费用和效果外，还有一个“齿距”属性：0、1或2。

例如：

| 卡牌 | 费用 | 效果 | 齿距 |
|---|---:|---|---:|
| 切沙刃 | 1 | 对单体造成7点伤害 | 1 |
| 折光盾 | 1 | 获得6点护甲 | 0 |
| 越级传动 | 0 | 下一张卡齿距+1，抽1张 | 2 |
| 逆转轮组 | 1 | 城环反向移动；本回合首次触发模块时再抽1张 | 1 |

卡牌结算后，城环指针按齿距移动。玩家选择卡牌时，不仅考虑即时伤害或防御，还要考虑它会把指针送到哪里。

### 2. 第二构筑轴：三段式“城环”

战斗界面中央下方始终显示一个三段圆环：

```text
[区段A] → [区段B] → [区段C] → 回到A
                 ↑ 当前指针
```

每个区段可安装一个模块。卡牌令指针移动；落到某一区段时，该模块触发。每个模块每回合默认只能触发一次，避免无限循环。

示例模块：

- 聚光塔：落点卡是攻击牌时，使目标“暴露”；下一次受到的伤害增加4。
- 折叠墙：落点卡是防护牌时，保留本次获得护甲的一半至下回合。
- 余热工坊：本回合第三次落到这里时，返还1点行动点，但下回合少抽1张。
- 双向轮组：首次触发后，下一张牌的移动方向反转。
- 沙晶透镜：齿距为2的牌落入本区段时，复制其非伤害效果一次。
- 空载仓：没有安装模块时无效果；连续两回合落入后生成一张临时“应急零件”。

### 3. 为什么它真正改变决策

玩家每次出牌会同时回答三个问题：

1. 这张牌当前能否解决敌人的攻击意图？
2. 它的齿距会把城环送到哪个模块？
3. 为了触发更强模块，是否值得先出一张即时收益较低的调度牌？

模块位置也构成决策：同样三个模块，排列顺序不同，适配的卡牌序列不同。奖励阶段需要在“补齐牌组”“安装新模块”“升级现有模块”之间取舍。

这不是被动数值饰品：它直接影响出牌顺序、目标选择、保留资源、奖励方向和删牌判断。

### 4. 定向构筑与信息反馈

- 战斗奖励先选类别：三选一卡牌，或二选一模块。
- 奖励权重略微偏向当前构筑缺少的齿距和相邻模块组合，但不会保证出现。
- 模块可在工坊移动位置或升级，预览不修改状态，确认后才扣资源。
- 卡面始终显示齿距。
- 选中卡牌时，城环预演目标落点；未确认前不移动。
- 模块说明、实际效果、奖励文本和档案条目全部读取同一份数据。

## 四、交互与状态架构

### 场景状态机

```text
title → map
map → battle → reward → map
map → workshop/event → map
map → boss → victory
battle/boss → defeat
```

存档必须直接保存 `scene` 及该场景所需的 `pendingState`，不能根据 DOM 猜测玩家所在页面。

### 状态分层

```ts
GameState {
  saveVersion: number
  profile: ProfileState
  run: RunState | null
  preferences: Preferences
}

ProfileState {
  unlockedCardIds: string[]
  discoveredEnemyIds: string[]
  discoveredModuleIds: string[]
  archiveEntries: string[]
  completedRuns: number
}

RunState {
  seed: string
  scene: Scene
  hp: number
  maxHp: number
  currency: number
  deck: CardInstance[]
  routeNodes: RouteNode[]
  routeLinks: RouteLink[]
  routeHistory: string[]
  currentLayer: number
  modules: ModuleSlot[]
  combat: CombatState | null
  pendingReward: RewardState | null
  pendingEvent: EventState | null
  flags: Record<string, boolean>
}

CombatState {
  turn: number
  energy: number
  hand: CardInstance[]
  drawPile: CardInstance[]
  discardPile: CardInstance[]
  enemies: EnemyState[]
  selectedCardId: string | null
  targetMode: "none" | "enemy" | "self" | "all-enemies"
  ringPointer: 0 | 1 | 2
  triggeredSlotsThisTurn: number[]
  phase: "player" | "resolving" | "enemy" | "won" | "lost"
}
```

### 出牌交互契约

```text
idle
  → 点击单体攻击牌：进入 enemy-targeting
  → 点击自用牌：进入 self-targeting 或直接显示自身确认
  → 点击群体牌：显示全部有效目标

targeting
  → 有效目标高亮，城环显示预计落点
  → 点击有效目标：只结算一次，支付费用，移动城环，清空选择
  → 点击无效目标：不消耗费用、不移动城环
  → 再点已选卡、点空白处、按 Escape：取消
  → 目标在确认前死亡：拒绝结算并清空目标
```

手机提供明确的“取消选择”按钮，不能只依赖 Escape。结束回合按钮在所有目标横屏视口中持续可见。

### 数据与纯逻辑边界

以下函数独立于界面，便于固定种子测试：

- `resolveCard(state, cardId, targetIds)`
- `previewRingLanding(state, cardId)`
- `triggerModule(state, slotIndex, context)`
- `generateRoute(seed)`
- `getReachableNodes(run)`
- `rollReward(seed, buildSnapshot)`
- `migrateSave(rawSave)`
- `resolveEnemyTurn(combat)`
- `serializeRun()` / `hydrateRun()`

## 五、开发顺序

1. 建立共享数据结构、场景状态机、种子随机和存档版本。
2. 用占位图完成标题→地图→战斗→奖励→地图→首领→结算。
3. 实现抽牌、弃牌、费用、伤害、护甲、敌方意图和多敌人目标状态机。
4. 实现三段城环、齿距移动、落点预览及3个首批模块。
5. 实现真实路线图：路线连线数据同时控制绘制和点击资格。
6. 实现卡牌/模块分类奖励、工坊预览与确认。
7. 扩展至12张卡、6个模块、2套普通遭遇和两阶段首领。
8. 加入 profile/run 分离、旧存档迁移、继续游戏。
9. 完成桌面与手机横屏布局、低高度模式和竖屏提示。
10. 加入原创视觉、状态动画、音效入口和数据驱动帮助。
11. 做对抗性检查、缓存隔离、公开部署和线上版本验证。

每一步都先通过自己的验收，不能等全部内容完成后一起排错。

## 七、可验证验收标准

| 编号 | 验收方法 | 通过条件 |
|---|---|---|
| A01 完整循环 | 新档连续游玩 | 无调试入口即可完成地图→两场战斗→奖励→首领→结算 |
| A02 双构筑有效 | 用相同牌组、更换模块排列重复固定种子战斗 | 至少出现两种不同的合理出牌顺序或资源结果 |
| A03 目标状态机 | 测试选中、有效目标、无效目标、取消、目标死亡、群体目标 | 无效与取消不扣费、不移环、不触发模块；结算不重复 |
| A04 多目标 | 双敌战分别攻击两个目标并使用群体牌 | 高亮、结算和死亡后目标资格正确 |
| A05 路线真实 | 检查至少20个种子 | 所有节点可达；点击资格与可见连线完全一致；不能跨线选择 |
| A06 奖励构筑 | 连续采样100次奖励 | 卡牌三选一不重复；模块二选一；领取后不能重复领取 |
| A07 说明一致 | 对比卡面、预览、档案及实际结算 | 全部来自共享数据；不存在文本与效果不一致 |
| A08 预览安全 | 在工坊反复移动、升级预览后取消 | 资源、模块位置和等级完全不变 |
| A09 存档隔离 | 胜利后新开一局 | profile解锁保留；上一局HP、牌组、路线和奖励不泄漏 |
| A10 旧档迁移 | 加载至少两个旧schema夹具 | 自动补全新字段且保留历史路线；迁移后可继续并再次保存 |
| A11 响应式 | 测试1366×768、844×390、780×430、568×320及低高度横屏 | 手牌、敌意图、城环、取消和结束回合可见；无阻断性纵向滚动；关键触控区约44px |
| A12 断点边界 | 每个媒体查询在阈值−1px、阈值、阈值+1px测试 | 无按钮消失、卡牌裁切、遮挡或布局跳入错误模式 |
| A13 竖屏 | 手机竖屏打开 | 显示明确旋转提示，不进入不可操作的半布局 |
| A14 重复动作 | 连点卡牌、目标、奖励、结束回合 | 单次输入最多结算一次；结算阶段屏蔽重复操作 |
| A15 缓存一致 | 发布新版本后以旧缓存、无痕窗口分别打开 | CSS/JS带版本号或内容哈希，不会混用新HTML与旧脚本 |
| A16 公开入口 | 从非开发设备打开规范URL | 无需登录或本地文件；HTML、CSS、JS、代表资源均返回成功 |
| A17 部署身份 | 对比页面版本、构建元数据和目标commit | 公开页面明确对应本次部署commit，而非旧构建 |
| A18 原创性 | 搜索素材来源、名称和页面文本 | 无未经许可素材、现有游戏专有文本、角色或界面复刻 |
| A19 首次可理解 | 让未读说明的测试者游玩10分钟 | 能独立选牌、选目标、取消、结束回合，并解释城环落点 |
| A20 固定种子 | 两次使用同一版本和种子开局 | 路线、遭遇和初次奖励一致，便于复现缺陷 |

第一版真正的完成线不是“能打开”，而是：陌生玩家能从规范公开链接完成一局，理解牌组与城环之间的关系，手机横屏不丢失主操作，旧档可迁移，线上资源与目标版本完全一致。
