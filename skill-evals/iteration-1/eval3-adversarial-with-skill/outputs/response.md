结论：“链接能打开”只证明服务器返回页面，不代表完整产品闭环可靠。本次只读检查发现以下 10 个互不重复、可复现的问题。

1. 首局没有教玩家如何结束回合｜中

- 复现：清空站点数据后首次进入。欢迎页只说明“点牌→点目标”；进入战斗、耗尽潮力后，没有首次回合引导解释“结束回合”。
- 玩家影响：新玩家会在第一次无牌可打时停住，误以为游戏卡死。
- 根因：首次教学只覆盖选牌和目标，没有覆盖完整回合闭环；[index.html](/C:/Users/SuperKing/Documents/PPT/spirelike/index.html:18) 与结束回合按钮之间没有引导状态。
- 修复：首次耗尽潮力或连续数秒无有效操作时，高亮“结束回合”，同时提示触摸与 `E` 键；完成一次后永久关闭。
- 通过条件：10 名未看说明的新玩家都能在不受外部指导的情况下完成第一个敌方回合。
- 旧玩家保护：新增 `onboardingVersion`，只对未完成该版本教学的档案展示，不强制老玩家重看。

2. 键盘和读屏用户无法按标准方式选择战斗目标｜高

- 复现：只按 Tab 导航。己方船虽然声明 `role="button"`，但没有 `tabindex`；敌舰是带 `onclick` 的 `<article>`，既不是按钮，也没有角色和焦点入口。
- 玩家影响：鼠标可玩的攻击、防御，在键盘和辅助技术下不可发现、不可聚焦；自定义方向键不能替代完整语义。
- 根因：[index.html](/C:/Users/SuperKing/Documents/PPT/spirelike/index.html:60) 和 [game.js](/C:/Users/SuperKing/Documents/PPT/spirelike/game.js:425) 把点击行为绑定在非原生交互元素上。
- 修复：目标改为原生 `<button>`，或补齐 `role`、`tabindex`、Enter/Space 行为、选中状态和目标说明；保留现有方向键快捷操作。
- 通过条件：仅用 Tab、Enter、Space 可完成攻击、防御和取消；读屏能朗读舰名、HP、意图、是否可选，击沉目标退出焦点序列。
- 旧玩家保护：不改变点牌→点目标的鼠标/触摸合同，只增加语义层和焦点管理。

3. 700/701px 竖屏断点存在一像素悬崖｜高

- 复现：分别使用 `700×900` 与 `701×900` 竖屏。前者出现旋转提示；后者提示完全消失，直接进入为横屏设计的地图/战斗布局。
- 玩家影响：微信内嵌浏览器、分屏和平板处在 701px 附近时，会得到拥挤或裁切界面，却没有横屏指引。
- 根因：[styles.css](/C:/Users/SuperKing/Documents/PPT/spirelike/styles.css:68) 将方向保护同时绑定到 `max-width:700px`，把“是否需要横屏”错误等同于“设备宽度不超过 700px”。
- 修复：按方向和可用纵横比判断，例如所有窄纵横比竖屏显示安全提示；同时使用 `100dvh`、safe-area 和 visual viewport 处理微信工具栏。
- 通过条件：699、700、701、768px 竖屏均出现可理解的安全入口；568×320、780×430、844×390 横屏不被遮挡。
- 旧玩家保护：纯布局修复，不改游戏状态；保持原横屏尺寸和触摸坐标规则。

4. 旧版本航程不会迁移，而是被当作无存档｜阻断级

- 复现：在 `localStorage` 中放入 v1 存档或旧键名，再打开 v0.12。加载器只读取 `wakebound-run-v2`，且要求 `version===2`，否则返回 `null` 并创建新航程。
- 玩家影响：已公开迭代多轮后，老玩家的路线、牌组、装备、待领取奖励或故事节点可能直接消失。
- 根因：[game.js](/C:/Users/SuperKing/Documents/PPT/spirelike/game.js:224) 和 [game.js](/C:/Users/SuperKing/Documents/PPT/spirelike/game.js:244) 采用严格版本拒绝，没有按版本逐步迁移，也不探测旧键。
- 修复：实现 `v1→v2→当前版` 的纯迁移链；迁移路线历史、奖励形状、故事、装备等级及当前场景；失败时保留原始备份并提供安全恢复。
- 通过条件：地图、战斗、奖励、事件、维修、船坞和故事七类旧存档夹具均恢复到同一逻辑场景；重复加载迁移结果一致。
- 旧玩家保护：先复制旧值再迁移，成功写入新键后才标记完成；绝不先删除旧键或静默开新档。

5. “三张不同稀有度”是错误承诺｜中

- 复现：连续生成卡牌奖励。`rollCardRewards()` 只做加权随机和卡牌去重，没有保证普通、罕见、稀有各一张，因此经常出现相同稀有度。
- 玩家影响：玩家根据页面/README 的承诺期待跨稀有度比较，实际奖励规则却不同，削弱构筑判断和信任。
- 根因：[game.js](/C:/Users/SuperKing/Documents/PPT/spirelike/game.js:251) 的候选规则与对外文案不是同一来源。
- 修复：若设计确实要求不同稀有度，采用三个明确稀有度槽位并为精英调整槽位概率；否则删除“不同稀有度”承诺，显示真实抽取规则。
- 通过条件：批量执行 10,000 次奖励测试，每组满足已声明规则、三张不重复、只能领取一张；旧的待领取候选不被重抽。
- 旧玩家保护：存档里已有 `pendingReward.cardIds` 原样保留，新规则只用于尚未生成的奖励。

6. 首领开局固定，重复游玩可以机械背板｜中

- 复现：多次抵达任意故事首领。普通敌人的起手意图由种子决定，但首领的 `move` 被固定为 `0`，所以同一首领永远用第一招开场。
- 玩家影响：三形态、多阶段看似丰富，实际开局可被固定脚本解决，削弱重复游玩价值。
- 根因：[game.js](/C:/Users/SuperKing/Documents/PPT/spirelike/game.js:301) 对 boss 使用 `node.type==='boss' ? 0 : ...`，绕过了已有的种子随机机制。
- 修复：为首领定义合法开局池或种子化开局模式，同时保证第一回合意图公开且不生成不公平组合。
- 通过条件：抽查至少 30 个种子，每个首领存在两个以上合法开局；相同种子完全可复现，意图与实际行动一致。
- 旧玩家保护：已进入首领战的存档保留当前 `move`；只对尚未生成的首领战应用新开局规则。

7. 故事选择写着“确认”，实际单击立即永久生效｜高

- 复现：进入任一故事章节，点击任一选项一次。金币、卡牌、最大生命和故事旗标立即修改，没有预览或第二次确认。
- 玩家影响：手机误触会永久改变后续精英战、船坞折扣、首领形态和结局，无法撤回。
- 根因：[game.js](/C:/Users/SuperKing/Documents/PPT/spirelike/game.js:392) 的按钮文案为“确认这个选择”，但 `onclick` 直接调用 `resolveStoryChoice()`。
- 修复：第一次点击只打开“选择结果与后续影响”预览，第二步才提交；提交期间锁定输入并原子保存。
- 通过条件：预览、返回、刷新均不改变状态；只有明确确认一次才写入旗标，快速双击也只能结算一次。
- 旧玩家保护：已完成章节不重放；旧的 `pendingStory` 缺少确认阶段时默认恢复到选择页，而不是自动提交。

8. 微信中曾导致敌舰消失的脆弱滤镜仍在运行｜高

- 复现：在部分微信 WebView 或较旧移动 WebKit 中进入战斗。舰名、HP、意图仍显示，但敌舰 PNG 可能因组合滤镜渲染失败而透明/缺失。
- 玩家影响：玩家看不到攻击目标，误以为资源未加载或游戏损坏。
- 根因：[styles.css](/C:/Users/SuperKing/Documents/PPT/spirelike/styles.css:202) 仍对敌舰图片使用 `hue-rotate + saturate + contrast + drop-shadow` 组合；这正是案例 C15 记录过的兼容性风险。
- 修复：移除舰体图片上的动态滤镜；差异化改用预生成变体、稳定的外框、徽章和独立阴影层；启动时预加载并检测图片尺寸。
- 通过条件：微信真机至少覆盖两代 iOS/Android WebView，所有敌舰图像可见；禁用 CSS filter 后仍能区分普通、精英和首领。
- 旧玩家保护：不改变敌人 ID、数值或存档结构，仅替换渲染方式，保留现有资源路径作为回退。

9. 所谓“永久装备”在新航程中全部清空｜高

- 复现：当前航程获得并升级硬件，然后选择重新启航或通关后开始新航程。`newRun()` 始终设置 `hardware:[]` 和 `hardwareLevels:{}`。
- 玩家影响：玩家投入金币形成的永久装备构筑消失，“永久装备”名不副实，也破坏长期成长预期。
- 根因：硬件仅存在于 run state；永久 profile 只保存 XP、已见卡牌和故事档案。[game.js](/C:/Users/SuperKing/Documents/PPT/spirelike/game.js:234) [game.js](/C:/Users/SuperKing/Documents/PPT/spirelike/game.js:289)
- 修复：明确产品语义。若确为永久装备，将装备等级纳入带版本的 profile，并在新航程初始化装备/装载方案；同时重新校准起始强度。
- 通过条件：通关、失败、主动重开、刷新和升级版本后装备均按规则保留；单局临时效果不会写入永久档案。
- 旧玩家保护：首次升级时把旧玩家当前合法硬件迁入永久档案，过滤未知 ID、限制 Lv.1–3，并保留迁移前备份。

10. 两个公开试玩链接返回的不是同一款版本｜阻断级

- 复现：分别访问 README 中两个公开地址。两者均返回 HTTP 200，但 GitHub Pages 显示 `v0.12.0`、加载 `game.js?v=20260715-3`，包含故事和硬件入口；另一个“公开试玩站”没有这些 v0.12 签名，页面结构和资源入口也不同。
- 玩家影响：一部分玩家拿到最新版，另一部分拿到旧版或另一套构建；反馈无法复现，缓存问题会被误判为玩法问题。
- 根因：两个公开 URL 由不同发布链路维护，没有共享发布产物、版本清单或上线后签名校验。“HTTP 200”被错误当成发布成功。
- 修复：指定一个 canonical 试玩地址；另一个地址保持可用但重定向，或部署完全相同的构建产物。发布后校验 HTML 版本、JS/CSS 标签、关键功能签名和文件哈希。
- 通过条件：两个旧链接仍能打开并最终得到同一版本；HTML、核心 JS/CSS 和代表资源均为 200，版本号及构建哈希一致，缓存更新后旧客户端自动切换。
- 旧玩家保护：不要废弃现有 URL；保留原入口并跳转到 canonical 地址，继续使用相同域名存档时还需提供 localStorage 导入/迁移方案。
