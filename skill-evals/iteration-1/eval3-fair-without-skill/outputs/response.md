以下是基于题目信息做的假设驱动对抗性测试设计。这里的“问题”指必须主动验证的高风险缺陷候选；不能因为链接可访问就视为功能正确。

## 1. 旧存档在新版本中被清空或无法读取

**复现步骤**

1. 准备上一版本存档，至少包含已解锁地图、永久装备、故事进度和当前战斗。
2. 更新到最新版。
3. 使用同一浏览器重新打开公开试玩链接。
4. 分别测试正常刷新、强制刷新、关闭浏览器后重开。
5. 观察是否从新手流程开始，或控制台出现字段缺失、JSON 解析错误。

**玩家影响**

长期进度、永久装备或故事解锁丢失。对公开试玩而言，这是最容易直接失去老玩家信任的问题。

**可能根因**

存档结构升级后直接按新字段读取，没有 `saveVersion`、默认值和逐版本迁移；或者初始化失败时把空白存档覆盖到了旧存档上。

**修复**

- 为存档增加明确的结构版本。
- 实现逐版本迁移，例如 `v1 → v2 → v3`，不要只维护“旧版到最新版”的一次性转换。
- 读取失败时保留原始存档，禁止自动覆盖。
- 对新增字段设置兼容默认值。
- 提供存档导出、备份和恢复入口。

**通过条件**

- 至少准备最近三个版本的真实存档夹具，全部能够升级并继续游玩。
- 升级前后的永久装备、货币、地图和故事进度一致。
- 迁移失败时显示恢复提示，原始存档仍然存在。
- 重复启动迁移不会重复发放或扣除资源。

**避免破坏旧玩家**

上线前对旧存档做副本迁移；新存档验证成功后才切换。旧数据至少保留一个版本周期，并允许回滚或导出。

## 2. 地图节点连续点击导致重复移动或无路可走

**复现步骤**

1. 进入存在多个相邻节点的地图。
2. 在移动动画尚未结束时，快速点击两个不同节点。
3. 再测试双击同一节点、点击后立即返回上一页、弱网下连续点击。
4. 检查角色位置、已访问标记、移动次数和可选节点是否一致。
5. 尝试继续进入下一场战斗。

**玩家影响**

可能一次消耗两次行动、跳过节点、进入逻辑上不相邻的位置，或地图显示有路但所有节点均不可点击，形成永久软锁。

**可能根因**

地图移动由多个点击事件并发修改；视觉动画、逻辑位置和存档位置分别更新；移动请求没有状态锁或幂等标识。

**修复**

- 地图移动期间锁定输入。
- 使用单一状态事务同时提交当前位置、路径和消耗。
- 在逻辑层再次验证目标节点是否与当前位置相邻。
- 动画只表现已提交的结果，不参与决定游戏状态。
- 载入存档时校验当前位置是否仍存在合法出口。

**通过条件**

连续点击、双击和弱网模拟下，每次只产生一次合法移动；重新载入后位置与画面一致；任何正常节点都不会让玩家进入无出口软锁。

**避免破坏旧玩家**

为已经落入非法节点的旧存档增加修复逻辑：自动回退到最近的合法节点，同时不重复扣除行动或重置整张地图。

## 3. 战斗结算被重复触发

**复现步骤**

1. 将敌人生命压到可以被一张牌击杀。
2. 快速连点该牌，或在伤害动画期间点击结束回合。
3. 在胜利动画出现前切到后台再返回。
4. 尝试浏览器后退后再前进。
5. 检查奖励、战斗次数、故事触发和地图解锁是否执行多次。

**玩家影响**

玩家可能重复获得奖励，也可能先进入胜利状态又被敌方行动击杀；严重时胜利弹窗与失败弹窗同时出现，无法继续。

**可能根因**

出牌、动画回调、敌人死亡监听和回合结束都能独立调用结算；缺少唯一的战斗状态机和“一次性结算”保护。

**修复**

建立明确状态：

`玩家回合 → 动作处理中 → 敌方回合 → 结算中 → 已结束`

只有状态机能够进入结算；结算使用唯一 `battleId`，奖励和胜负写入均按该 ID 幂等处理。进入“结算中”后立即禁用所有战斗输入。

**通过条件**

在连点、切后台、回退、降帧和动画跳过条件下，每场战斗只能产生一个最终结果和一份奖励；胜负状态不能相互覆盖。

**避免破坏旧玩家**

不要仅根据“敌人生命为零”重新补发奖励。读取旧存档时同时检查 `battleId` 和已结算记录；无法确定时优先恢复到结算确认页，而不是直接重复发奖。

## 4. 卡牌奖励可以重复领取或被永久吞掉

**复现步骤**

1. 战斗胜利后进入三选一卡牌奖励。
2. 点击一张卡的同时快速双击或点击其他卡。
3. 选择后立刻刷新页面。
4. 在奖励页使用浏览器返回，再进入。
5. 分别检查牌库中卡牌数量和奖励页状态。

**玩家影响**

可能获得两张甚至多张奖励，破坏平衡；也可能奖励页已标记完成但卡牌没有加入牌库，导致玩家永久少一次成长机会。

**可能根因**

“加入卡牌”和“标记奖励已领取”分两次保存；点击没有锁；奖励没有唯一 ID；刷新后依据页面路由而不是存档事务恢复。

**修复**

- 每次奖励生成唯一 `rewardId`。
- 选择动作采用原子事务：校验未领取、加入卡牌、记录选择、关闭奖励。
- 第一次点击后立即锁定所有选项。
- 重载时根据 `rewardId` 恢复到未选择或已完成状态。
- 服务端存在时增加幂等键；纯本地游戏也要保存已消费 ID。

**通过条件**

任意连点、刷新、返回操作下，一个 `rewardId` 最多增加一张卡；未完成的奖励能恢复；已完成奖励不能重新领取。

**避免破坏旧玩家**

对旧存档中没有 `rewardId` 的奖励只生成一次稳定 ID。不要重新扫描历史胜场并补发奖励，否则会给老玩家造成重复资产。

## 5. 永久装备在读档后重复叠加属性

**复现步骤**

1. 记录无装备时的基础生命、攻击或费用上限。
2. 获得并装备一件永久装备。
3. 刷新页面并重复进入游戏三次。
4. 尝试卸下再装备、替换同槽装备、开始新局。
5. 对比每次载入后的最终属性。

**玩家影响**

属性可能每刷新一次就增加，导致数值失控；相反，若卸下逻辑重复扣减，也可能出现负属性或老玩家突然变弱。

**可能根因**

存档同时保存“已修改后的最终属性”和装备列表，读档时又执行一次装备加成；装备效果使用增量修改，而非从基础值重新计算。

**修复**

- 存档只保存基础成长与装备身份，不保存可推导的最终属性。
- 最终属性通过“基础值 + 当前装备 + 永久升级”纯函数计算。
- 装备、卸下和读档都调用同一套重算逻辑。
- 为唯一装备增加实例 ID，禁止同一实例重复生效。

**通过条件**

连续刷新、反复装备和开始新局后，属性始终等于规则计算结果；同一装备只生效一次；卸下后精确恢复，不产生累计误差。

**避免破坏旧玩家**

迁移时不能简单把旧最终属性当成新基础属性。应根据旧装备反推出基础值；无法可靠反推的存档要保留快照，并提供一次性兼容补偿或人工恢复方式。

## 6. 故事事件被跳过、重复播放或卡住主流程

**复现步骤**

1. 到达一个会触发故事的地图节点。
2. 在对话逐字播放时刷新。
3. 快速连续点击“下一句”。
4. 在最后一句出现时切后台或返回地图。
5. 再次进入该节点，检查故事是否重复、跳过或阻断后续节点。

**玩家影响**

玩家可能错过关键剧情，也可能每次进入地图都被迫重看；若故事完成标记未写入，还可能无法解锁下一节点。

**可能根因**

故事进度仅记录“是否看过”，没有记录事件、段落和完成状态；剧情播放与地图解锁分别保存；多个点击跳过了中间副作用。

**修复**

- 使用稳定的 `storyEventId` 和段落索引。
- 区分“未开始、播放中、已完成”。
- 只有明确完成事件后才执行一次解锁副作用。
- 快速点击只能推进一段，不能绕过状态提交。
- 重载后从最近已确认段落恢复，并提供跳过已读内容的功能。

**通过条件**

刷新每个段落、快速点击和反复进入节点均不会丢失剧情或重复执行奖励；完成事件后主流程必然解锁；已读剧情可回看但不会重复改变状态。

**避免破坏旧玩家**

保留旧版“已看过”标记，并将其映射为新事件的完成状态。新增剧情不要复用旧事件 ID，否则老玩家可能被误判为已读。

## 7. 写存档途中关闭页面会产生不可恢复的坏档

**复现步骤**

1. 在战斗结算、获得装备、切换地图等保存密集时刻刷新或关闭标签页。
2. 使用浏览器开发工具模拟低性能和存储写入延迟。
3. 模拟存储空间不足、隐私模式或本地存储被禁用。
4. 重新打开游戏。
5. 检查是否白屏、无限加载或自动创建新档覆盖坏档。

**玩家影响**

一次异常关闭即可损坏完整进度；如果系统自动初始化，坏档还可能覆盖仅有的恢复证据。

**可能根因**

直接覆盖主存档；写入过程非原子；没有校验和、备份槽和异常恢复；保存失败没有向玩家反馈。

**修复**

采用两阶段保存：

1. 写入临时槽。
2. 完整读取并校验。
3. 将当前主存档轮换为备份。
4. 再切换临时槽为主存档。

同时保留若干最近快照，捕获配额和权限异常，并明确提示玩家。

**通过条件**

在任意关键保存点强制终止后，重新打开至少能恢复到当前或上一次有效状态；损坏数据不会导致白屏；保存失败不会显示“已保存”。

**避免破坏旧玩家**

首次启用新机制时先复制旧存档为只读备份，确认新格式成功写入后再使用新主槽；绝不在解析失败时自动删除旧键。

## 8. 音乐重复播放、无法静音或切后台后仍占用资源

**复现步骤**

1. 进入首页后主动开启音乐。
2. 多次在地图、战斗、奖励和故事页面间切换。
3. 快速返回上一页或重复进入同一场景。
4. 切到微信后台一分钟再返回。
5. 调整静音后刷新，检查是否恢复用户设置。

**玩家影响**

多个音轨叠加造成爆音；静音设置失效会在公共场合突然播放；后台持续播放或重复创建播放器会增加耗电和内存占用。

**可能根因**

每个页面挂载时都创建新音频实例，但卸载时没有停止；路由切换重复注册监听；未处理浏览器自动播放限制和页面可见性事件。

**修复**

- 全局只保留一个音频管理器。
- 场景切换复用实例并执行可取消的淡入淡出。
- 页面隐藏时根据用户设置暂停，恢复时只恢复一次。
- 音乐必须由首次用户交互解锁。
- 音量、静音和是否允许后台播放单独持久化。

**通过条件**

连续切换场景数十次仍只有一个活动音轨和一组监听器；静音刷新后保持；微信切后台再回来不会叠播；未交互前不会触发自动播放报错循环。

**避免破坏旧玩家**

迁移已有音量字段并尊重原设置；新版本默认不要擅自把曾经静音的玩家重新设为有声。

## 9. 微信横屏下画面可见但关键按钮无法操作

**复现步骤**

1. 在微信内置浏览器打开链接。
2. 分别在刘海屏、圆角屏和普通手机上旋转为横屏。
3. 测试从竖屏进入后再旋转，以及启动时直接横屏。
4. 打开和关闭微信顶部控件、系统手势栏或软键盘。
5. 点击屏幕边缘的卡牌、结束回合、设置和关闭弹窗按钮。

**玩家影响**

按钮可能被安全区、微信控件或手势栏遮住；画面缩放后触摸坐标错位，玩家看得到卡牌却点不中；旋转后还可能只显示半屏。

**可能根因**

使用固定 `100vh`、固定像素画布或启动时缓存的宽高；未监听方向和可视视口变化；渲染缩放与触摸坐标换算使用不同尺寸；未处理安全区。

**修复**

- 基于 `visualViewport` 和实际容器尺寸重新布局。
- 监听方向变化与视口 resize，并在稳定后重算一次。
- 渲染和命中测试共享同一个坐标变换。
- 使用 `env(safe-area-inset-*)` 保留安全区。
- 关键操作按钮不得紧贴屏幕边缘。
- 横屏提示只作为辅助，不能遮挡已经横屏的玩家。

**通过条件**

目标微信机型矩阵中，旋转前后所有关键按钮均完整可见且命中准确；连续旋转不会累积缩放；没有横向滚动条、半屏黑区或弹窗无法关闭。

**避免破坏旧玩家**

保留桌面端和已适配机型的现有断点，把微信修复限制在能力检测或对应媒体条件中；不要用微信 User-Agent 强制一套固定分辨率。

## 10. 发布后新旧资源混用，链接能开但游戏不可继续

**复现步骤**

1. 先访问旧版本并保持标签页开启。
2. 发布包含脚本、卡牌数据或素材文件改名的新版本。
3. 在旧标签页刷新、强制刷新和离线后恢复网络。
4. 再用新标签页、微信内置浏览器和浏览器缓存访问。
5. 检查是否出现旧 HTML 加载新脚本、脚本请求旧数据、素材 404 或版本不一致提示。

**玩家影响**

首页可能正常打开，但进入战斗后白屏、卡图缺失、规则数据不匹配；老玩家尤其容易因缓存命中旧入口而遇到问题。

**可能根因**

静态资源没有内容哈希；HTML 被长期缓存；Service Worker 缓存策略不完整；发布过程非原子；代码、配置和存档迁移没有共享版本号。

**修复**

- JS、CSS、数据和素材使用内容哈希文件名。
- HTML 短缓存或必须重新验证。
- 每次构建生成统一 `buildId`，运行时验证代码与数据版本一致。
- 先完整上传新资源，最后切换入口文件。
- Service Worker 使用版本化缓存，并在安全时提示玩家刷新。
- 保留旧版本资源一段时间，避免仍在线的旧客户端立即失效。

**通过条件**

旧标签页、新标签页、微信缓存和弱网情况下都只能加载一套完整兼容的构建；所有资源请求成功；版本不匹配时出现可恢复提示，而不是白屏。

**避免破坏旧玩家**

不要发布后立刻删除旧哈希资源；先保证旧客户端能完成当前战斗并保存。强制刷新前必须提交存档，并明确告知更新不会清除进度。

## 建议的放行标准

公开试玩至少满足以下条件后，才能从“链接可打开”升级为“可以稳定交给玩家”：

- 旧存档迁移、重复结算和坏档恢复测试全部通过。
- 地图、战斗、奖励、装备和故事的关键操作均经过连点、刷新、返回、切后台测试。
- 微信横屏完成真机测试，而不只是桌面浏览器模拟。
- 发布过程验证新旧缓存混用。
- 每个版本保留可恢复的旧存档和旧静态资源。
- 对阻断流程、丢档、重复发奖、无法操作四类问题实行零容忍。
