审查边界：遵照要求，我没有查看源码、README、Git、线上链接或原始对话，也没有修改文件。以下是根据题面与 Skill 质量门禁设计的**假设驱动、可证伪的对抗性问题清单**；按复现步骤出现对应症状，即可确认缺陷。

## 1. 首局主操作在低高度横屏中不可发现

**严重度：P0，首局阻断**

- **复现：**清空站点数据，以新玩家身份用微信内置浏览器打开；横屏设为 `568×320` 或 `844×390`，进入首场战斗，打完可用牌后寻找“结束回合”。
- **症状：**结束回合按钮被手牌、浏览器工具栏或安全区遮挡；或按钮存在但与次要按钮同级，玩家不知道下一步。
- **玩家影响：**首局无法继续。“链接能打开”但核心循环并不可玩，玩家会直接退出。
- **根因：**主要操作依赖普通文档流或只按宽度适配，没有按实际可视高度、浏览器栏和手牌抬升空间布局；把“控件存在”误当成“玩家能发现”。
- **修复：**将结束回合固定在稳定操作层；低高度模式保持可见、约 `44×44px` 以上；首次战斗增加一次性指引，同时保留触摸和键盘路径。
- **通过条件：**在 `568×320、780×430、844×390、1024×600` 中，无需滚动即可看到并点击；10 名首次玩家均能在10秒内结束第一回合。
- **旧玩家保护：**用已持久化的 `onboardingSeen` 跳过新手提示；不更改回合规则、快捷键和存档结构，只调整控件层级。

## 2. 选牌—选目标状态可被连点、取消或死亡目标破坏

**严重度：P0，战斗状态损坏**

- **复现：**多敌人战斗中点击一张攻击牌；快速双击第二个敌人，或在敌人死亡动画期间再次点击；再测试空白处取消、切换卡牌、按返回键以及点击无效目标。
- **症状：**伤害落到第一个敌人、费用扣两次、同一张牌结算两次、无效目标也消耗能量，或死亡目标重复播放击沉效果。
- **玩家影响：**玩家无法信任战斗输入，可能因此输掉一局；触屏双击尤其容易触发。
- **根因：**DOM 点击事件直接修改战斗数据，缺少单一的“待选牌/合法目标/正在结算”状态机与幂等锁；动画状态和逻辑状态混在一起。
- **修复：**建立显式 targeting 状态；结算前再次验证目标仍存活且合法；结算期间锁输入；资源只在成功结算后扣除；击沉动画记录一次性 transition 标志。
- **通过条件：**对同一目标连续触发10次点击，只结算一次；无效或死亡目标不扣费；取消后没有残留箭头或高亮；多目标测试始终命中所选对象。
- **旧玩家保护：**维持现有“点牌→点目标”的交互合同，不改卡牌数值；载入旧战斗存档时清除无法迁移的临时 targeting 状态，但保留牌库、血量、能量和敌人状态。

## 3. 微信横屏在断点相邻尺寸重新发生遮挡

**严重度：P0，移动端不可玩**

- **复现：**依次测试 `779、780、781、844、850、851px` 宽度，并改变浏览器工具栏显示状态；打开地图、战斗手牌、奖励弹窗和设置弹窗。
- **症状：**某个精确宽度正常，宽一像素却恢复桌面 HUD；地图被按钮覆盖、手牌顶部被裁切、奖励确认按钮落到屏幕外，或横向滚动意外产生纵向滚动条。
- **玩家影响：**同型号手机因微信版本、系统缩放或浏览器栏差异得到完全不同的可玩性。
- **根因：**响应式规则只验证单个截图尺寸，断点区间不连续；依赖布局视口而不是实际 `visualViewport` 高度。
- **修复：**按宽度、高度和安全区建立状态矩阵；把紧凑模式覆盖至完整风险区间；弹窗允许安全滚动；手牌预留选中抬升空间；触控目标保持约44像素。
- **通过条件：**所有断点上下各1像素，以及题列7种代表视口中：主操作不重叠，HP、货币、进度仍可见，弹窗确认/取消均可达，无非预期纵向滚动条。
- **旧玩家保护：**只调整 CSS/布局状态，不变更 DOM 数据键、按钮语义和玩法；保留旧玩家的字号、动效等显示偏好。

## 4. 多轮迭代后的旧存档无法安全迁移

**严重度：P0，旧玩家资产风险**

- **复现：**准备多个旧版本存档，分别停在地图、战斗、卡牌奖励、故事选择、装备奖励和最终结果页；用新版本载入，再测试字段缺失、旧奖励结构和一段损坏 JSON。
- **症状：**白屏、无限刷新、回到错误场景、奖励重复领取、路线历史失效、故事标志丢失、永久装备清空，或损坏存档让游戏永远无法启动。
- **玩家影响：**长期玩家丢失进度和永久资产，是最严重的信任破坏之一。
- **根因：**直接把当前状态结构反序列化，没有 `saveVersion`、逐版本迁移、默认值和损坏隔离；把永久档案与单局状态混存。
- **修复：**增加版本化迁移链；分离 profile 与 run；迁移必须纯函数化、可重复执行；迁移前备份原存档；无法恢复时提供安全重开单局而不是清空永久档案。
- **通过条件：**所有历史夹具都载入到正确逻辑场景；迁移执行两次结果相同；旧路线、故事、装备、牌组不丢；损坏存档不会循环崩溃，并可导出备份。
- **旧玩家保护：**永不原地覆盖唯一副本；保留旧存档键至少一个发布周期；记录迁移版本，支持回滚读取；任何字段补齐均使用不增益、不减益的安全默认值。

## 5. 地图看似分叉，实际点击资格与连线不一致

**严重度：P1，核心策略失真**

- **复现：**连续生成至少100个种子；每层选择不同分支，比较画出的连线、真正可点击节点和下一层可达集合；重点检查故事、精英、维修、商店与 Boss 前一层。
- **症状：**所有节点实际都能通向下一层；画面没有线却能点击；有线却不能进入；某些未来节点没有入口，或路线能生成死路。
- **玩家影响：**地图选择只是装饰，玩家无法规划血量、金币、故事和装备路线；严重时整局无法完成。
- **根因：**绘制连线与点击资格分别计算，或者只根据层号判断可达，没有共享的显式 `routeLinks` 图结构。
- **修复：**生成种子化有向图；渲染、预览和点击统一读取同一组边；生成时验证每个未来节点有入边、每个当前节点可继续且 Boss 可达。
- **通过条件：**1000个种子全部可完成；至少存在真实的单边约束；点击集合与显示边集合完全一致；没有孤立节点和死路。
- **旧玩家保护：**已开始的旧局保留历史路线和当前节点；仅为尚未访问的未来层补边，禁止重排已展示或已选择节点；新图算法只应用于新局。

## 6. 内容调度没有“记忆”，战斗和故事连续重复

**严重度：P1，重玩价值坍塌**

- **复现：**连续完成3局，并在同一局进入相邻的普通战斗和故事节点；记录敌人编队、战场条件、开局模式、故事章节及选项回调。
- **症状：**同一敌群连续出现，区别只有血量增加；故事节点反复抽到相同章节；故事选择只改即时文本，后续战斗、Boss形态或故事档案没有任何变化。
- **玩家影响：**地图虽有很多节点，但玩家感受到的是重复关卡；故事选择也失去意义。
- **根因：**内容从单一随机池独立抽取，没有近期历史排除、章节资格、故事 flag 和下游回调；精英/Boss只是普通敌人数值放大。
- **修复：**增加带历史约束的内容调度器；战斗按敌群、条件和开局模式组合；故事选择写入持久 flag，并由后续节点、奖励或 Boss 读取。
- **通过条件：**相邻节点不出现完全相同组合；样本种子达到预设的敌群/条件覆盖率；每个关键故事选择至少有一个可自动验证的后续差异；精英与 Boss 有独立机制问题。
- **旧玩家保护：**不重写当前局已经抽出的事件和战斗；旧局缺失历史字段时从已完成记录推导，无法推导则使用空历史；既有故事选择不被重新解释或清除。

## 7. 卡牌奖励与装备说明无法支持真实选择

**严重度：P1，构筑决策失真**

- **复现：**赢得战斗进入卡牌奖励；对三张候选牌逐项检查费用、稀有度、流派、升级结果和关键词；再用触屏查看永久装备的触发条件、等级和实际效果。
- **症状：**三张牌只是同效果换数值；说明与战斗计算不一致；触屏无法打开 tooltip；装备只写氛围文案，不说明触发、叠加或槽位；领取后实际效果与奖励页不同。
- **玩家影响：**玩家只能盲选，错误归因于自己；卡牌奖励和永久装备无法形成可规划的双构筑轴。
- **根因：**奖励文字、图鉴、升级预览和战斗逻辑分别维护；只做了 hover 信息面，没有触摸、键盘和辅助技术入口。
- **修复：**卡牌和装备使用单一规则数据源；奖励固定呈现多名不同候选且只允许选一项；同一数据派生短说明、完整说明、升级预览、图鉴和实际结算。
- **通过条件：**所有奖励文本与确定性规则测试一致；触摸、键盘和鼠标均能查看详情；候选至少在费用、用途或流派上形成真实差异；一次奖励最多提交一次。
- **旧玩家保护：**先统一描述，不静默改变旧卡牌/装备机制；如必须修正规则，使用内容版本号并在旧局中保持旧版本效果，新局才启用新版。

## 8. 微信浏览器的音乐启动与场景切换失败

**严重度：P1，核心体验缺失**

- **复现：**微信内首次打开，未点击任何按钮时观察音频；点击开始后依次进入地图、普通战、精英、Boss和奖励；切到后台再返回，测试静音、音量和来电中断恢复。
- **症状：**自动播放被拒绝后永远静音；多个场景曲目叠播；后台恢复后音量翻倍；静音图标与真实状态不一致，或用户偏好刷新后丢失。
- **玩家影响：**作者以为“已经有音乐”，真实玩家却无声、爆音或被叠播打扰。
- **根因：**没有把音频解锁绑定到明确用户手势；场景路由创建多个播放器；未处理播放 Promise、页面生命周期和持久偏好。
- **修复：**首屏保留明确的“开始游戏/启用声音”手势；使用单一音频管理器；场景切换先淡出并停止旧轨；正确处理拒绝、暂停、恢复和持久音量。
- **通过条件：**首次用户手势后音乐可靠启动；任意时刻最多一条背景轨；六类场景路由正确；后台往返20次无叠播；静音和音量刷新后保持。
- **旧玩家保护：**沿用现有音量与静音存储键并迁移数值范围；没有明确选择的旧玩家默认保持其原有静音状态，不能升级后突然外放。

## 9. 永久装备效果在载入、新局或升级时重复叠加

**严重度：P0，永久进度与平衡损坏**

- **复现：**记录无装备基础属性；装备一件永久装备后保存，连续刷新或载入5次；再开始新局、升级该装备至最高级并重复领取同一装备。
- **症状：**每载入一次属性再次增加；单局临时加成带回永久档案；最高级仍累计经验但显示错误；重复装备被无限转化为数值，导致强度失控。
- **玩家影响：**部分旧玩家意外变得极强，另一部分可能因迁移丢失加成；排行榜、难度和奖励价值全部失真。
- **根因：**装备效果通过“在当前属性上加值”应用，而不是从基础属性和装备清单重新派生；缺少等级上限、重复领取和 profile/run 边界。
- **修复：**把装备 ID、等级和槽位作为唯一持久真相；每次载入从基础值重算派生属性；效果应用幂等；最高级明确显示 `MAX`，重复奖励按公开规则处理。
- **通过条件：**连续载入100次属性不变化；新局只带永久效果、不带临时效果；最高级不溢出；卸下/换装后结果与从零重算完全一致。
- **旧玩家保护：**迁移时保留装备所有权和合法等级；对异常累计属性以装备清单重算，但先备份旧值并生成修复记录；不要删除无法识别的旧装备，先隔离为待兼容项。

## 10. 公网链接返回200，但玩家加载到混合版本缓存

**严重度：P0，发布真实性失败**

- **复现：**先在微信打开旧版本并缓存；发布新版后不清站点数据，分别普通刷新、重新进入聊天链接和离线恢复；核对 HTML、JS、CSS、资源版本及存档迁移。
- **症状：**HTML 是新版、JS 或 CSS 仍是旧版；按钮存在但事件不存在；新存档结构被旧脚本读取后崩溃；分享出去的是仓库页而不是直接试玩页。
- **玩家影响：**作者本机正常，旧玩家却随机白屏或布局错乱；“链接能打开”产生错误完成感。
- **根因：**只检查入口 HTTP 200，没有验证部署提交、资源签名、缓存标识和旧缓存升级路径；发布物不是原子版本。
- **修复：**保留稳定试玩 URL，但为 CSS/JS/关键资源加发布版本；入口带构建签名；资源清单同版本发布；若使用 Service Worker，采用可回滚的更新策略并避免新旧壳混用。
- **通过条件：**公开 HTML、JS、CSS和代表资源均返回200且构建 ID 一致；部署提交与目标提交相同；旧缓存设备升级后不会混载；直接试玩链接可从微信进入完整循环。
- **旧玩家保护：**保持原 URL、localStorage/IndexedDB 命名空间和迁移入口；先部署能读取新旧存档的兼容代码，再切换资源版本；至少保留上一稳定构建用于紧急回滚。

这10项分别覆盖：首局发现性、输入状态、响应式边界、存档迁移、地图策略真实性、战斗与故事重复、卡牌奖励与信息面、音乐及微信兼容、永久装备进度、公开部署与缓存。任何一项未经对应通过条件验证，都不足以用“链接能打开”判断游戏已经可公开交付。
