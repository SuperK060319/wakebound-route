“链接能打开”只证明首页资源能返回，不代表核心循环、状态一致性、兼容性和旧存档安全。以下按公开试玩版进行黑盒对抗审查；每项都给出可重复的测试路径，实际验收时应逐条留存录像、控制台日志和存档快照。

### 1. 地图节点可被重复结算

**复现**

1. 进入任意奖励型地图节点。
2. 完成事件或领取奖励。
3. 在结算动画期间连续双击节点、快速刷新，或使用浏览器后退再前进。
4. 重新进入地图，观察金币、卡牌、装备是否重复增加，节点是否仍可点击。

**玩家影响**

玩家可以刷取奖励；也可能出现奖励增加但节点未完成，或节点完成却没收到奖励。排行榜、难度和平衡数据都会失真。

**根因**

节点完成、奖励发放和地图推进由多个异步操作分别提交；前端没有点击锁，存档也缺少“本节点已结算”的幂等标识。

**修复**

以唯一的 `runId + nodeId + resolutionId` 作为结算键。一次事务内完成奖励写入、节点完成和下一节点解锁；按钮首次触发后立即锁定，重复请求直接返回首次结果。

**通过条件**

同一节点在双击、刷新、后退、断网重连等情况下最多结算一次；存档中的资源变化、节点状态与结算记录完全一致。

**保护旧玩家**

旧存档缺少结算键时，不要重置地图。根据已访问节点、当前路径和奖励记录生成兼容键；无法判断的存档只标记当前节点为待确认，不回收玩家已有资源。

### 2. 战斗快速操作导致重复出牌或负费用

**复现**

1. 进入战斗，保留一张费用刚好等于当前能量的卡。
2. 快速双击卡牌，或在拖拽出牌时同时点击结束回合。
3. 在低帧率设备、CPU 限速或弱网模拟下重复。
4. 检查卡牌效果是否执行两次、能量是否为负、卡牌是否同时出现在手牌和弃牌堆。

**玩家影响**

可能无成本多次触发伤害，也可能吞牌、卡死回合或造成不可继续的战斗状态。

**根因**

动画层和规则层都能触发出牌；费用校验与效果执行不在同一原子操作中，结束回合也没有等待出牌事务完成。

**修复**

所有战斗动作进入单一命令队列。规则层只接受一次带 `actionId` 的出牌命令，并按“校验—扣费—移区—执行效果”顺序同步提交；动画只消费结果，不修改战斗状态。

**通过条件**

连续点击、拖拽冲突和结束回合竞争下，每张牌只结算一次；能量不为负；任何时刻一张卡只能存在于一个区域。

**保护旧玩家**

战斗中旧存档若没有动作队列版本，应从最近一个完整回合边界恢复，而不是强行解析半完成动画；恢复前保留原存档备份，并补偿本场已消耗的入场资源。

### 3. 状态效果结算顺序不稳定

**复现**

1. 构造同时具有护盾、持续伤害、回合末回血、易伤或反伤的战斗。
2. 在同一回合触发击杀与持续伤害，随后结束回合。
3. 记录正常速度结果，再开启后台标签页、切换页面或降低帧率重复。
4. 比较敌人死亡、玩家受伤、回血和奖励触发顺序。

**玩家影响**

相同操作可能产生不同结果，例如敌人死亡后仍攻击、护盾提前清零、持续伤害未结算或双方同时死亡却判定不同。

**根因**

规则结算依赖动画回调、计时器或对象遍历顺序，没有明确的阶段模型和效果优先级。

**修复**

建立固定阶段：出牌结算、死亡检查、回合结束效果、再次死亡检查、回合切换。效果按稳定优先级和唯一序号排序，禁止用动画完成事件驱动规则。

**通过条件**

同一种子、同一牌序和同一操作序列重复运行至少100次，前后台切换及不同帧率下最终状态完全一致。

**保护旧玩家**

给新规则增加战斗规则版本。旧玩家正在进行的战斗继续使用旧规则或安全回退到本场战斗开始；不要在半场战斗中切换结算顺序。

### 4. 卡牌奖励可重复领取或跳过后仍入库

**复现**

1. 赢得战斗并进入三选一卡牌奖励页。
2. 连续点击两张卡，或点击卡牌后立即点击“跳过”。
3. 在选择瞬间刷新页面。
4. 检查牌组、奖励页状态和地图推进。

**玩家影响**

玩家可能获得两张牌、获得卡牌同时拿到跳过补偿，或完全丢失奖励并被卡在结算页。

**根因**

选牌、跳过和页面跳转是独立事件，没有互斥状态；奖励候选也可能在刷新后重新随机生成。

**修复**

生成奖励时立即保存不可变的 `rewardId` 和候选列表。领取接口只接受一次决策：某个候选或跳过；成功提交后才能离开页面。

**通过条件**

双击、跨按钮点击、刷新和重连后，单个 `rewardId` 始终只有一个最终决策；候选列表不会变化；流程不会卡死。

**保护旧玩家**

旧存档若停留在奖励页，先固化当前可见候选。若无法恢复候选，提供一次明确标注的补偿选择，不自动塞入随机卡牌，也不改动既有牌组。

### 5. 永久装备在新周目中重复叠加或丢失

**复现**

1. 获得一件永久增加初始生命、金币或抽牌数的装备。
2. 结束当前周目并开启新周目，记录初始属性。
3. 连续刷新、退出登录再进入，并再次开始新周目。
4. 卸下、升级或重复获得同名装备后继续测试。

**玩家影响**

永久加成可能每次加载都重复叠加，导致属性无限增长；也可能新周目不生效或装备直接消失，破坏长期成长价值。

**根因**

存档保存的是“已经加成后的最终属性”，加载时又重新应用装备；或把永久库存错误地放在单局存档中。

**修复**

分离基础属性、永久库存和单局临时修正。最终属性始终通过纯函数派生；永久装备使用稳定实例 ID，并明确唯一、可叠加和升级规则。

**通过条件**

同一存档加载10次属性不变化；新开周目只应用一次永久效果；装备增删前后的属性差值严格符合配置。

**保护旧玩家**

迁移时不要简单从现有总属性中盲减装备值。保留原存档快照，利用历史版本、装备清单和合理上限推导基础值；有歧义时选择“不削弱玩家”的结果并记录一次性迁移标记。

### 6. 故事事件条件与实际状态不一致

**复现**

1. 首次进入关键故事节点并完成一种选择。
2. 刷新后重进节点，或在另一条地图路径绕回相关事件。
3. 分别携带和不携带指定卡牌、装备或关键道具。
4. 检查已读剧情是否重复、选项是否错误解锁、后续文本是否承认先前选择。

**玩家影响**

玩家会看到角色“失忆”、死亡角色再次出现、未满足条件却解锁选项，甚至因缺少剧情标记而无法继续。

**根因**

剧情展示状态、剧情完成状态和分支选择混用一个布尔值；条件读取展示层数据，而不是权威存档状态。

**修复**

拆分 `seen`、`resolved`、`choice` 和 `consequenceApplied`。故事选择使用事件 ID 幂等提交；条件只读取稳定的游戏状态和版本化剧情标记。

**通过条件**

刷新与绕路不会重复发放剧情效果；每个选择只产生一次后果；所有后续台词与先前选择一致；不存在无法退出的故事页。

**保护旧玩家**

旧存档缺少分支标记时，根据已获得道具、已解锁节点和任务状态回填。不能可靠推断时，提供中性兼容分支，不撤销奖励、不强制玩家重演长剧情。

### 7. 存档写入竞争造成回档或损坏

**复现**

1. 在战斗胜利、卡牌奖励和地图返回连续发生时立刻关闭标签页。
2. 同时在两个标签页打开游戏并分别推进。
3. 模拟存储空间不足、隐私模式、浏览器清理站点数据或写入异常。
4. 再次打开，检查能否加载以及加载到哪个时间点。

**玩家影响**

数小时进度可能回退；更严重时整个存档无法解析，只能重新开始。多标签页还可能让较旧状态覆盖较新状态。

**根因**

直接覆盖单一存档；写入不是原子的；没有校验和、版本号、递增修订号、备份槽及并发写入检测。

**修复**

采用“写临时记录—校验—切换主指针”的原子保存方式，保留最近若干快照。每次保存包含 schema 版本、revision、时间戳和校验和；检测到多标签页版本冲突时阻止旧 revision 覆盖新 revision。

**通过条件**

在任意保存阶段强制关闭100次，均能恢复到最后一个完整检查点；损坏主存档时自动读取备份；两个标签页不能静默互相覆盖。

**保护旧玩家**

加载器必须先复制旧存档再迁移，迁移失败则继续保留旧数据并允许导出。新版本首次启动不可自动覆盖唯一的旧格式存档。

### 8. 版本更新后旧存档字段不兼容

**复现**

1. 准备多个历史版本存档：局外界面、地图中、战斗中、奖励页和故事页各一份。
2. 升级到当前版本。
3. 逐份加载并继续至少经过一次节点结算和一次保存。
4. 检查未定义数值、空白卡牌、缺失装备、地图断路和循环报错。

**玩家影响**

老玩家可能更新后白屏、卡牌变成空对象、地图无法继续，或者系统把“不认识的字段”当成零值并永久覆盖。

**根因**

存档结构随代码直接变化，缺少 schema 版本和逐级迁移；卡牌或装备使用易变化的名称、数组位置作为标识。

**修复**

为存档定义显式 schema，提供 `vN → vN+1` 的连续迁移和校验。内容实体使用永不复用的稳定 ID；删除内容时保留兼容占位符与补偿策略。

**通过条件**

所有受支持历史版本均能加载、继续游戏并再次保存；迁移前后核心资源守恒；未知内容不会导致崩溃或整份存档清空。

**保护旧玩家**

迁移永远在副本上执行并保留原始数据；无法映射的旧卡牌或装备应转为等价物或补偿选择，不静默删除。提供存档导出和回退入口。

### 9. 音乐初始化、切换和静音状态失控

**复现**

1. 在手机或桌面浏览器首次打开游戏，不进行任何点击。
2. 进入战斗、返回地图、触发故事，再快速切换标签页。
3. 反复切换音乐开关并刷新。
4. 检查是否无声、重复叠播、切场景爆音，以及静音设置是否保留。

**玩家影响**

音乐可能完全无法播放，也可能多个场景音乐同时叠加；玩家静音后刷新又突然响起，公开试玩时尤其容易造成直接退出。

**根因**

没有遵守浏览器自动播放限制；每个页面或场景自行创建音频实例；静音值只改界面图标，没有统一控制音频总线或持久化。

**修复**

首次用户手势后统一解锁音频上下文；使用单例音频管理器管理场景曲目、淡入淡出和可见性变化；把静音与音量保存为用户设置。

**通过条件**

首次未交互时不报错、不强制播放；首次点击后可正常启用音乐；任意场景最多一个主音乐实例；静音刷新后仍有效；前后台切换无叠播。

**保护旧玩家**

旧存档没有音频设置时默认采用保守音量或静音提示，不把游戏进度迁移与音频设置绑定；更新后不得擅自覆盖玩家已有静音偏好。

### 10. 微信横屏下关键控件不可见或触摸坐标偏移

**复现**

1. 使用微信内置浏览器打开游戏。
2. 从竖屏旋转为横屏，再旋回竖屏并再次横屏。
3. 在有刘海、圆角、底部手势条的设备上进入地图、战斗和奖励页。
4. 展开微信顶部/底部控件、触发键盘或退出再返回。
5. 点击贴近屏幕边缘的卡牌、结束回合和确认按钮。

**玩家影响**

按钮可能被安全区或微信工具栏遮住；视觉位置与实际点击区域不一致；卡牌拖拽失效，玩家无法结束回合或领取奖励。

**根因**

只在首次加载时读取视口尺寸；依赖固定 `100vh` 或设计稿像素；未处理 `visualViewport`、安全区、设备像素比和旋转后的坐标重算。

**修复**

监听 `resize`、`orientationchange` 和 `visualViewport`；使用动态视口单位及 `safe-area-inset-*`；所有命中检测基于当前画布变换矩阵。无法进入合格横屏时显示明确引导，但不能让游戏处于半可操作状态。

**通过条件**

主流 iOS、Android 微信环境在首次横屏和多次旋转后，所有核心按钮完整可见且可点击；触摸命中位置与视觉元素一致；地图、战斗、奖励和故事都能完成全流程。

**保护旧玩家**

横屏适配只能改变显示和输入映射，不应写入战斗、地图或牌组状态。旋转期间暂停交互并保留当前操作上下文，恢复后不能重复出牌或重复结算。

### 发布门槛

这10项不应只做“手点一次”的冒烟测试。公开试玩重新宣称“没问题”前，至少需要满足：

- 为每项保存固定复现脚本、测试存档和预期结果。
- 核心结算加入幂等与状态不变量检查。
- 覆盖刷新、双击、后退、多标签页、断网、低帧率和强制关闭。
- 建立历史存档样本库，任何版本发布前自动跑迁移。
- 微信真机覆盖至少一台 iOS 和一台 Android，不用桌面模拟器代替。
- 上线前先灰度给一小部分旧玩家；检测迁移错误率、战斗中断率和存档恢复率。
- 所有存档升级先备份、后迁移、再校验；失败时可回滚，绝不以“新存档能玩”替代旧玩家兼容性验收。
