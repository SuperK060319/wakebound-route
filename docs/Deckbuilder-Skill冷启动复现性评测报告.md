# Deckbuilder Web Game Product Loop：冷启动复现性评测报告

## 结论先行

这套封装已经证明能稳定复现**方法水平**，但不能被夸大为“一条提示词自动复刻完整 v0.12 游戏”。

- 首轮三类公平评测中，有 Skill 的形式通过率为 **100%**；无 Skill 基线平均 **65.6%**。
- 最难的异主题类比任务进行了两次 Skill 重复采样，两次均为 **100%**，最低值 100%，形式方差 0。
- 初稿有执行可靠性问题：长任务第一次运行时，两个 Skill 样本和一个基线样本都没有输出最终答复，需要补发一次“立即输出”。
- Skill 加入短案例索引和规划输出预算后，新会话在首个回合直接交付，耗时由需要补救的约 13–17 分钟下降到约 3分43秒。
- 第二轮为了变短遗漏缓存破除，被严格判为 6/7；再加入五项不可压缩门槛后，第三轮首回合达到 7/7，耗时约 4分16秒。
- 一组评测因 Skill 会话查看了源码、基线却被禁止查看而作废。它被保留，但没有进入分数。

因此目前合理判断是：

> Skill 能让无原对话历史的新会话，更稳定地做出完整垂直切片、原创第二构筑轴、状态/迁移、目标交互、响应式边界、公开部署和缓存验收；它显著减少“只会给泛泛建议”的概率。它仍不能替代真实玩家对乐趣、审美和数值的判断。

## 1. 评测目的

本次不是验证 Skill 能否复述《潮痕航路》，而是回答三个问题：

1. 去掉 48 轮对话上下文后，能否保留关键工作方法？
2. 同一困难任务重复运行，是否仍能达到相同最低水平？
3. 换成《沙海机城》这一异主题项目，是否能迁移方法而不复制航海、锈蚀、潮力和原项目表达？

## 2. 隔离方法

所有执行样本都在新建的独立 Codex 任务中运行。新任务不继承本对话内容。

- **With Skill**：只获得评测题、Skill 路径和 Skill 路由要求的引用文件。
- **Without Skill**：获得相同题目，但被禁止读取 Skill、教程和《潮痕航路》源码。
- **公平 Eval 3**：两边都被禁止检查源码、Git、网络、README 和原始对话，只做假设驱动的测试设计。
- **重复采样**：异主题任务对 Skill 初稿独立运行两次。

没有用本窗口替新线程代答。输出、时长、评分证据和作废原因均保存到 `skill-evals/`。

## 3. 评测题

### Eval 1：响应式断点与安全重开

给定“780px 正常、812px 遮挡、重开紧贴静音且无确认”，要求诊断、最小修复和验收。

主要检查：断点两侧、44×44、保留 HP/金币/进度、低高度视口、症状—根因—修复—通过条件。

### Eval 2：异主题同水平类比

从零规划原创《沙海机城》，要求完整垂直切片、状态架构、真正改变决策的双构筑、开发顺序、提示词、手机横屏和公开交付。

明确禁止使用航海、锈蚀、潮力和参考游戏受保护表达。

### Eval 3：十问题对抗性审查

对一个有地图、战斗、奖励、永久装备、故事、存档、音乐和微信支持的公开网页游戏，提出十个互不重复的可复现问题。

主要检查：十个质量域、旧存档、真假路线/奖励选择、缓存部署、稳定直玩链接，以及“不把 HTTP 200 当完成”。

## 4. 首轮结果

| 评测 | 有 Skill | 无 Skill | 主要差异 |
|---|---:|---:|---|
| Eval 1 响应式 | 5/5，100% | 4/5，80% | Skill 明确保留 HP、金币和航程；题目本身线索多，区分度弱 |
| Eval 2 异主题 A | 6/6，100% | 3/6，50% | 基线缺具体迁移、多目标验证、断点±1、目标提交和缓存破除 |
| Eval 2 异主题 B | 6/6，100% | — | 第二个 Skill 样本再次满分 |
| Eval 3 公平审查 | 6/6，100% | 4/6，66.7% | 基线擅长事务安全，但遗漏真假路线/奖励质量和 canonical 直玩链接 |

聚合结果：

- 有 Skill：100% ± 0%。
- 无 Skill：65.6% ± 15.0%。
- 形式通过率提升：约 34.4 个百分点。

这里的“100%”只表示预先定义的形式门槛全部满足，不等于游戏一定好玩或美术一定优秀。

## 5. 偶然性与重复采样

### 内容稳定性

两个独立 Skill 样本都把《沙海机城》设计为原创机械城市，并形成了“指令牌组 × 城市轨环/脉冲模块”的第二构筑轴。两者都具备：

- 完整地图—战斗—奖励—Boss 闭环；
- 多目标点击状态机；
- profile/run 分离和迁移；
- 真实路线边；
- 低高度/断点测试；
- 目标提交、公开链接和缓存验收；
- 清晰不做清单。

两次形式通过率均为 100%，说明最低水平稳定。输出长度分别约 7,127 与 11,102 字，说明表达规模仍有明显随机波动。

### 执行稳定性

初稿的三个 Eval 2 会话（Skill A、基线、Skill B）第一次都没有产生最终答复，必须再发送一次“立即输出”。这说明长任务的失败不只来自 Skill，但 Skill 的多文件路由会加重风险。

这项失败没有因为最终内容通过而被删除。

## 6. 评测污染与作废处理

第一次 Eval 3 中，Skill 会话自行读取了当前项目源码和两个线上站点，基线却被禁止查看源码。虽然该 Skill 输出找到了真实问题，但两边输入权限不同，不能用于衡量 Skill 增益。

处理方式：

1. 原样保留在 `iteration-1/eval3-adversarial-*`；
2. 写明作废原因；
3. 新建公平 Eval 3，两边都只能看题目；
4. 基准只使用公平样本。

这是评测体系本身必须具备的对抗性审查。

## 7. Skill 迭代

### 初稿

初稿要求新项目读取 memory、architecture、prompt library，并在需要案例时读取完整案例库。内容质量高，但读取链路过长。

### 第二版：短案例索引 + 输出预算

调整：

- 新增 `case-index.md`；
- 新项目只读案例索引、架构和提示词库；
- 只有明确复刻历史偏好时才读 memory；
- 规划任务目标 2,500–5,000 字；
- 要求同一任务回合直接交付。

结果：首回合直接交付，223.376 秒；但漏写具体 CSS/JS 缓存破除，得分 6/7。

### 第三版：五项不可压缩门槛

新增规划输出必须明确包含：

1. profile/run + 旧存档迁移；
2. 选牌/目标/取消/无效/多目标；
3. 低高度 + 断点前后 1px；
4. canonical 直玩链接 + 目标 commit；
5. CSS/JS 版本号或内容哈希，避免新旧缓存混载。

结果：首回合 256.071 秒，7/7，输出 6,598 字。

## 9. Skill 产生的真实增益

最稳定的增益不是文采，而是以下项目更难被遗漏：

- 把参考操作写成可取消、可验证的状态机；
- 让第二构筑轴改变出牌顺序和目标，而不是被动加数值；
- 把故事、奖励、装备、路线做成真实选择；
- 分离 profile/run 并设计迁移；
- 检查断点前后 1px 和低高度；
- 不用 HTTP 200 冒充玩法或视觉正确；
- 保护稳定直玩链接、目标提交和缓存身份；
- 用案例说明为什么，而不只列一堆 MUST/NEVER。

## 10. 仍然不能保证的部分

- 不能保证生成的完整游戏自动达到《潮痕航路》48轮迭代后的全部细节。
- 不能自动证明“好玩”；仍需要玩家测试和数值迭代。
- 不能替代用户选择音乐、美术和叙事气质。
- 不能读取或封装 Claude/Codex 的隐藏系统提示词和私人思维链。
- 不能把一次满分评测解释成所有未来题目都稳定满分。

## 最终判断

Skill 已达到“可复用、可解释、可冷启动复现”的交付标准：

- 有完整案例库而非只有限制词；
- 有项目记忆、架构、提示词和质量门槛；
- 有可执行审计脚本；
- 有公平基线；
- 有重复采样；
- 有异主题类比；
- 有作废样本；
- 有失败后迭代；
- 最终新会话首回合 7/7。

复现的是一套“把模糊愿望编译成公开产品”的方法，而不是把某一次模型偶然性包装成神话。
