# 从一句“复刻杀戮尖塔”到《潮痕航路》v0.12

> 一套给人类和机器共同学习的完整制作教程。它记录的是可观察、可验证的过程：48 个任务轮次、20 次 Git 提交、玩家截图、参考视频、代码和测试。它不声称包含任何模型的隐藏思维链或 Claude/Codex 的私有系统提示词。

## 0. 最终结果

《潮痕航路》从一句“我想复刻一个杀戮尖塔”发展成了公开可玩的原创航海牌组构筑 Roguelike：

- 40 张卡牌、四种构筑流派；
- 八层真实分支航线；
- 普通战、精英战、三种故事首领和四种海况；
- 卡牌三选一、金币和船体硬件奖励；
- 硬件升级、组合联动和船厂购买；
- 药剂、经验、设置、键盘与触控；
- 分支故事、后续影响和跨局故事档案；
- 存档、旧存档迁移、微信横屏适配；
- 六类场景配乐、海盗音效与音乐鸣谢；
- GitHub Pages 公开链接和单文件便携版；
- 九组自动回归测试。

公开试玩：<https://superk060319.github.io/wakebound-route/?v=20260715-3>

这不是一次提示词“抽卡”抽出来的结果。它是一个连续编译过程：

```text
模糊愿望
  ↓ 冻结最小可玩闭环
可玩原型
  ↓ 用户真实操作
交互纠偏
  ↓ 视频/截图/玩家反馈
界面、画风、兼容性和内容迭代
  ↓ 市场对标
构筑深度、创新系统和故事
  ↓ 对抗性审查
10个结构性问题
  ↓ 每项独立实现与回归
公开 v0.12
```

## 1. 为什么这个结果不能归因于一条提示词

结果由四种上下文共同产生：

1. **用户品味的显式训练**：用户不断说“不对”“看不见”“像三个游戏”“没有更新”，把模糊好坏转成了具体标准。
2. **长期记忆的浓缩**：例如“单屏”“微信能玩”“直接链接”“防误触”“故事要有后果”“逐项修改”“要对抗性审查”。
3. **仓库的累积状态**：每次改动都站在前一版真实代码、素材、存档和部署之上。
4. **工作流的累积**：分析参考 → 冻结范围 → 实现 → 测试 → 发布 → 真实反馈 → 反向审查。

如果只把最后一条“把十个问题都修好”交给一个没有前文的新模型，它不知道：

- “正确攻击操作”已经被用户定义为点牌、箭头、点目标；
- “公开”不是仓库，而是点击即玩的页面；
- 用户的核心创新偏好是船体硬件双构筑；
- 玩家来自微信手机，低高度横屏是一级场景；
- 哪些功能已经存在，哪些判断曾经失败；
- 什么叫“看起来像同一个游戏”；
- 旧玩家正在使用链接，不能随便破坏存档。

因此复刻方法必须同时封装 **memory、案例、架构、提示词、测试和发布纪律**。

## 2. 第一性原则：把反馈编译成产品规则

这次最重要的方法不是“多写代码”，而是把自然语言反馈翻译成可验证合同。

| 用户原话 | 表面要求 | 编译后的产品规则 | 验收方式 |
|---|---|---|---|
| 每回合结束点哪里 | 加按钮 | 主操作必须永久可见且显著 | 首次玩家不需说明；低高度仍可见 |
| 一个屏幕要装下 | 缩小 | 主要战斗信息和操作不得纵向滚动 | 多视口无主控遮挡 |
| 没有地图，不知道进度 | 加地图 | 进度与下一选择必须可见 | 地图独立场景 + HUD 航程 |
| 不是拖过去，是点牌再点敌人 | 改操作 | 输入状态机必须匹配参考和用户心智 | 多敌目标、取消、无效目标测试 |
| 不能只是标志，要船和特效 | 换图 | 角色状态必须由动作反馈表达 | idle/attack/shield/hit/sink |
| C盘别人打不开 | 上传 | 交付必须是跨设备直接入口 | 公网 HTTP 200 + 直接链接 |
| 三个奖励像三个游戏 | 美化 | 同级组件共享尺寸、层级和操作 | 等宽等高、短屏无滚动 |
| 强化前要知道结果 | 加预览 | 昂贵/不可逆操作先预览再确认 | 预览不扣钱、不改状态 |
| 锈蚀有什么用 | 加说明 | 机制出现处必须有一致解释 | 卡牌、状态、tooltip、牌典同源 |
| 链接能打开所以没问题？ | 再测试 | 可访问性不等于玩法质量 | 对抗性审查分域找十个问题 |

这个表就是人类与机器都能复用的“反馈编译器”。

## 3. 完整制作步骤

### 第 1 步：把“复刻”改写成“原创垂直切片”

原始提示：

```text
我想复刻一个杀戮尖塔
```

先做三个判断：

1. 参考的是核心循环和信息结构，不复制原作美术、文本、角色、卡牌或代码。
2. 第一版目标不是商业体量，而是浏览器能打通一局。
3. 必须在一个独立目录开发，不污染工作区其他项目。

第一版冻结为：单角色、12 张原创牌、两类普通遭遇、一个首领、能量/抽牌/弃牌/护甲/状态、敌人意图、战后奖励、三层流程。

可复制提示词：

```text
请把[参考游戏]拆成最小完整闭环，但所有表达原创。先写本轮范围、明确不做什么和验收标准，再实现。验收是无需安装、浏览器打开即可完整打通一局。
```

为什么有效：它把“像某个大游戏”从无底洞变成了可以证伪的一次迭代。

### 第 2 步：先做完整闭环，不先做漂亮首页

第一版顺序：

1. 卡牌/敌人数据；
2. 抽牌、能量、出牌、弃牌；
3. 敌人意图和敌方回合；
4. 胜负；
5. 奖励；
6. 下一战和首领；
7. 最后才做视觉框架。

关键经验：一张卡的说明、升级、牌典和实际效果，应来自同一个数据源；否则后面每加一种界面都可能写出不同规则。

### 第 3 步：用第一次试玩暴露信息架构

首个反馈不是数值，而是“每回合结束点哪里”。这说明核心循环存在，但信息架构失败。

修复不是告诉玩家“右下角”，而是：

- 固定结束回合；
- 提高对比；
- 让窄屏也保留；
- 后续增加 `E` 快捷键。

然后用户提出“一屏装下”，于是把战场高度、手牌、顶部栏和低高度模式一起重构，而不是继续叠页面。

### 第 4 步：把地图从装饰变成场景

最初常驻顶部进度图解决了“不知道打到哪里”，但真正的爬塔选择需要地图独立场景。

于是形成稳定流程：

```text
航线地图 → 选择节点 → 战斗/事件 → 奖励 → 返回地图
```

这个场景拆分同时解决了单屏空间冲突。

### 第 5 步：用视频学习结构，不凭印象模仿

用户提供完整参考视频后，分析维度不是“它很好看”，而是：

- 顶部状态栏持续显示什么；
- 手牌平时和悬停时怎样变化；
- 敌人意图何时出现；
- 回合切换如何反馈；
- 战斗、奖励、牌组查看如何切场景；
- 玩家实际输入序列是什么。

第一次仍然判断错了攻击输入，做成拖拽。用户纠正后，真正合同变为：

```text
点击攻击牌 → 卡牌保持选中 → 箭头跟随 → 敌人高亮 → 点击指定敌人结算
```

同时补齐群怪，因为只有多个敌人时，目标选择才不是假功能。

### 第 6 步：从符号美术升级到状态美术

用户说“不能只有标志”，说明主题没有进入动作层。

统一视觉方向：深靛海、锈铜、骨白旧纸、手绘暗黑航海、粗糙墨线。然后把状态映射成动画：

- 常态：随波浮动；
- 攻击：前冲、武器轨迹；
- 防御：潮汐盾和水纹；
- 受击：后坐、闪光、火花；
- 击败：倾斜下沉；
- 地图：航海图和移动船标。

后续“沉船动画反复播放”暴露了一个架构问题：动画不能只看 `hp <= 0`，必须记录 `alive → sinking → sunk` 的一次性转换。

### 第 7 步：把攻击操作的语法复制给防御

防御牌一点击就立即生效，玩家感觉不到“套到自己的船”。修复为：

```text
点防御牌 → 蓝色箭头 → 自己的船高亮 → 点船 → 升盾动画
```

这是一个通用设计原则：概念对称时，交互最好也对称；不同只用于表达真正的策略差异。

### 第 8 步：把本地原型变成真正产品入口

交付经历了四层：

1. 本地 `index.html`：开发入口；
2. 单文件 `潮痕航路-Demo.html`：内嵌脚本、样式和资源；
3. GitHub 仓库：源代码；
4. GitHub Pages：别人点击即玩。

关键纠偏：

- `C:\...` 不是分享链接；
- 仓库首页不是游戏入口；
- 微信可能缓存旧 CSS/JS，所以要版本参数；
- 手机浏览器音频必须在“开始航行”点击后播放；
- 每次发布要确认 Pages 对应的提交，而不是只看到 `git push` 成功。

### 第 9 步：用主题创造第二构筑轴

用户提出“奖励不一定是卡牌，也可以升级船的硬件”，这是从换皮走向原创系统的转折点。

硬件系统逐步包含：

- 卡牌/金币/硬件的奖励类别；
- 左上角长期硬件栏；
- 鼠标悬停和手机点击说明；
- 三级升级；
- 舰炮/船壳/动力/设备等类别；
- 硬件组合；
- 船厂强化与购买；
- 两件候选；
- 优先补齐现有组合缺件。

为什么“组合缺件优先”重要：如果玩家八站都凑不出组合，系统只存在于设计文档里，不存在于玩家决策里。

### 第 10 步：统一奖励组件，而不是统一颜色

截图中卡牌奖励被旧 `.card` 样式压窄，金币和硬件又各用一套结构。修复方式：

- 三项共享同一个组件；
- 固定类型、视觉、标题说明、按钮四段；
- 同尺寸同基线；
- 只用颜色和图形区分类别；
- 桌面 196×258，短屏 196×180；
- “放弃战利品”仍在首屏。

这是前端一致性的可测定义，而不是“看着差不多”。

### 第 11 步：所有高成本操作先预览再确认

用户担心误触强化、拆解和重开。形成统一规则：

- 强化：展示强化前后与价格，确认后扣费；
- 拆解：不可逆警告 + 二次确认；
- 重开：移入设置 + 二次确认；
- 预览阶段不修改牌组、不扣金币。

### 第 12 步：每个机制配一个信息表面

用户问“锈蚀有什么用”，不是玩家理解力问题，而是规则藏得太深。

同一机制需要四层：

1. 卡牌牌面短说明；
2. 敌人状态短标签；
3. hover/touch 完整 tooltip；
4. 航海牌典完整词条。

再加入公开牌池、已发现/未发现、当前持有、已强化数量，以及三个药剂槽。长期硬件也有固定 HUD 和说明。

### 第 13 步：设置必须真的控制游戏

参考截图包含大量设置。没有照抄 Steam 控制器和多人选项，只移植与网页单机版相关的行为：

- `1–8` 选牌；
- 方向键切牌/切目标；
- `Enter` 确认；
- `Esc` 取消/设置；
- `E` 结束回合；
- `M/D/A/S/X/Space` 查看真实地图、牌组、牌堆和状态；
- 加速、抖动、精简特效、牌堆计数、音量；
- 设置持久化。

真实测试不是检查页面上有没有键位文字，而是选择第二张牌、切到第二个敌人、确认后检查该敌人生命值。

### 第 14 步：音乐先筛授权，再让用户选审美

先定义循环槽位：标题、地图、普通战、精英、首领、奖励/船厂/牌典。胜利、开箱、升级等用短音效。

每个槽位筛五首，记录：试听、许可证、商用/修改、署名、循环、体积和判断。用户选择后再接入。这样把任务拆成：

- 客观门槛：合法、能循环、体积合适；
- 主观判断：哪首最符合游戏。

最终六首曲目与授权写入设置和 README。

### 第 15 步：从直线关卡升级到随机航线和存档

市场对标后先做最影响重复游玩的三项：随机分叉、非战斗节点、自动存档。

第一版分叉后来被对抗性审查判定为“假分叉”：每个节点都能去下一层所有节点。最终修复：

- 节点与边分离；
- `routeLinks` 同时控制绘图和点击；
- 每个来源至少有后路；
- 每个未来节点有入口；
- 存量路线历史迁移时强制保留；
- 用种子复现。

存档则区分永久档案和单局状态，并支持在地图、战斗、奖励、事件、宝藏、维修、船厂和故事中恢复。

### 第 16 步：内容扩张围绕流派和敌人问题

不是盲目堆几百张牌，而是从 12 张扩到 40 张并建立四个流派：炮击、锈蚀、装甲、潮汐。然后让精英、首领和硬件与这些流派发生关系。

故事也不是四个奖励弹窗：

- 选择记录为单局 flag；
- 跨局 archive 永久记录；
- 后续章节文案回调；
- 精英锈蚀、开局抽牌、维修、船厂折扣等玩法影响；
- 序章誓言选择三种首领形态；
- 结局组合读取主要 flag。

### 第 17 步：真实玩家截图优先于本机自信

两个远程问题：

1. 敌船图片在某些微信环境消失，只剩名字、血条和尾迹。说明整体 DOM 正常，脆弱的是图片滤镜层。去掉不稳定滤镜并预加载。
2. 手机浏览器工具栏压缩高度，欢迎页按钮被裁。增加低高度紧凑模式、安全滚动和可达按钮。

这说明“我电脑上正常”和“真实用户可玩”是两种证据。

### 第 18 步：做一次真正的对抗性审查

用户要求“从游戏里找出十个问题”。第一次审查得到：

1. 手机 HUD 遮挡；
2. 新局清空故事档案；
3. 故事选择无后果；
4. 战后只有一张随机牌；
5. 每战免费恢复 12 HP；
6. 地图是假分叉；
7. 战斗重复；
8. 卡牌和敌船辨识度不足；
9. 硬件组合过度看运气；
10. 满级经验仍显示 `290/150`。

用户要求一点一点改。第一项修完后再次对抗检查，发现 780px 正常但 781–850px 又坏了；按钮只有 34×34；重开无确认；金币和航程被隐藏。最终测试范围扩到 568–1101px，按钮 44×44，保留生命/金币/航程，重开移入设置。

核心教训：测试某个视口不是测试响应式系统。断点前后 1px 才是最危险的地方。

### 第 19 步：把十个问题变成九组回归测试

新增 `tools/game_logic_smoke.mjs`，用 DOM/localStorage stub 加载原生浏览器脚本，分组测试：

- `story`：选择回调、战斗影响、boss 形态；
- `reward`：三张不同候选、稀有度、只拿一张；
- `healing`：基础 0、硬件/组合/剧情来源；
- `route`：真实限制、全图可达、旧路线保留；
- `combat`：敌组、精英、海况、开局行动和 boss 变化；
- `visual`：40 个卡牌身份、23 个敌舰身份；
- `hardware`：二选一、联动倾向、购买和扣费；
- `xp`：满级 `MAX`、无幽灵升级；
- `compatibility`：旧奖励、旧硬件、旧航线、故事档案迁移。

再加三层发布检查：

1. JS/CSS/资源/单文件静态完整性；
2. 本地 HTTP 200；
3. GitHub Pages 对应提交、线上 HTML/JS/CSS 版本和关键函数签名。

### 第 20 步：只有公开版本验收后才停止

最终提交 `ef1ed96` 部署成功，公共资源返回 200，v0.12 特征在线，工作区干净。此时才把十项标记完成。

## 5. 给机器使用的总提示词

把下面提示词与 Skill 一起给一个没有本对话历史的新会话：

```text
请读取 deckbuilder-web-game-product-loop Skill，并按它的案例驱动工作法处理这个项目。

目标：[项目目标]
当前文件：[路径]
参考材料：[截图/视频/链接]
平台：[桌面/手机横屏/微信]
公开入口：[有则提供]

先做四件事：
1. 重建真实需求和后来纠正过的交互合同；
2. 从案例库选择至少两个最接近的案例，说明可迁移的是根因还是表面；
3. 定义本轮最小改动与可证伪验收；
4. 实现、回归、旧存档和公开链接验证。

不要复制参考游戏的受保护表达；不要用HTTP 200冒充视觉和玩法正确；无法做真实浏览器检查时要明确标注。
```

## 6. 给人类使用的操作法

人类不需要会写代码，但需要提供高价值判断：

1. **先给目标和约束**：主题、平台、是否公开、是否移动端。
2. **试玩时说具体动作**：不要只说“不好”，说“点防御就立即触发，我感觉不到它套到船上”。
3. **截图时保留环境**：设备、浏览器、横竖屏、地址栏和问题发生前一步。
4. **主观内容给选择**：音乐、美术方向让机器先筛合法候选，人类最终判断。
5. **要求对抗性审查**：当机器说“都正常”时，要求它从不同质量域找十个可复现问题。
6. **要求逐项关闭**：每项都要失败条件、修复、测试和公开验收。
7. **保留失败案例**：失败不是噪声，它是下一版 Skill 最值钱的案例。

## 7. 如何验证这套方法不是偶然

必须进行三类冷启动测试：

### A. 有 Skill 与无 Skill 对照

同一个新会话任务，一份加载 Skill，一份不加载。比较是否能：

- 把反馈翻译成状态合同；
- 使用案例而非泛泛建议；
- 给出断点、旧存档和发布验收；
- 避免只做表面美化。

### B. 同提示词重复采样

最难任务至少跑两次有 Skill 版本。看平均、最低值和方差，而不是只展示最好的一次。

### C. 同水平异主题类比

例如把航海换成《沙海机城》。合格结果应该迁移：垂直切片、双构筑、状态架构、质量门槛和公开流程；不应该机械复制潮力、锈蚀、船体或八层。

具体评测已封装在：

- `skills/deckbuilder-web-game-product-loop/evals/evals.json`
- `skills/deckbuilder-web-game-product-loop/references/eval-protocol.md`

## 8. 哪些东西不能封装

1. **隐藏系统提示词或思维链**：不可见，也不应伪造。
2. **用户临场审美判断**：Skill 可以筛选和解释，但不能保证每次音乐、美术选择都与用户相同。
3. **运气归零**：模型输出存在波动，只能通过案例、脚本、重复采样和最低门槛降低。
4. **真实玩家体验**：自动测试不能替代首次玩家试玩。
5. **商业级内容量**：方法能稳定生产结构，不代表一轮就能自动产生几百张平衡卡牌。

## 9. 最终检查清单

### 产品

- [ ] 一局闭环可打通；
- [ ] 操作语义与用户纠正一致；
- [ ] 原创差异改变玩家决策；
- [ ] 机制有解释、预览和确认；
- [ ] 路线、奖励和故事选择是真选择。

### 技术

- [ ] 状态与 DOM 分离到足够可测试；
- [ ] 永久档案与单局存档分开；
- [ ] 旧存档迁移；
- [ ] 一次性动画不会重复；
- [ ] 多种子和重复动作已测试。

### 手机与公开

- [ ] 多宽度、多高度和断点前后 1px；
- [ ] 44×44 左右触控目标；
- [ ] 微信兼容与缓存版本；
- [ ] 直接游玩链接而非仓库；
- [ ] 部署提交和公网资源已核对。

### 复用性

- [ ] Skill 有案例库，不只有规则；
- [ ] 有架构和提示词；
- [ ] 有可执行脚本；
- [ ] 有冷启动基线、重复采样、异主题类比；
- [ ] 报告最差样本和未通过项，不只展示冠军样本。

## 结论

这次真正可复刻的，不是《潮痕航路》的某个 CSS 或某条长提示词，而是：

> 用最小闭环获得真实反馈，把每次纠偏编译成产品合同和回归测试，再用原创差异、兼容性和公开交付把原型逐层变成产品。

当 memory、案例库、工作流、测试和失败边界一起封装后，新会话才有机会达到相近水平；只有流程框架和大量“必须/禁止”，无法复现这个结果。
